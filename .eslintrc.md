# ESLint Configuration

This document describes the ESLint configuration for the Faces of Plants monorepo.

## Overview

The project uses a shared ESLint configuration at the root level that is extended by each package. This ensures consistent code quality and style across all packages while allowing package-specific customizations.

## Configuration Structure

```
faces-of-plants/
├── eslint.config.mjs           # Root shared configuration
├── packages/
│   ├── core/
│   │   └── eslint.config.mjs   # Core package config (extends root)
│   ├── functions/
│   │   └── eslint.config.mjs   # Functions package config (extends root)
│   └── web/
│       └── eslint.config.mjs   # Web package config (extends root + Next.js)
```

## Root Configuration

The root `eslint.config.mjs` provides:

### TypeScript Best Practices
- Warns on explicit `any` types
- Enforces consistent type imports
- Warns on unused variables (with `_` prefix exception)
- Prefers `const` over `let`
- Warns on non-null assertions

### Import Ordering
Imports are automatically organized in the following order:
1. Node.js built-in modules (`fs`, `path`, etc.)
2. External packages (`react`, `zod`, etc.)
3. Internal workspace packages (`@faces-of-plants/*`)
4. Parent directory imports (`../`)
5. Sibling directory imports (`./`)
6. Index file imports
7. Type imports

Within each group, imports are alphabetized.

### Code Quality Rules
- No `console.log` (warnings, allows `console.warn` and `console.error`)
- No `debugger` statements
- Prefer template literals over string concatenation
- Enforce strict equality (`===`)
- Require curly braces for all control statements
- Consistent semicolons and quotes

## Package-Specific Configurations

### Core Package (`packages/core`)
- **Stricter type checking**: `any` types are errors (not warnings)
- **Explicit return types**: Functions should declare return types
- Used for shared business logic and utilities

### Functions Package (`packages/functions`)
- **Console allowed**: Lambda functions use console for CloudWatch logs
- **AWS globals**: Recognizes AWS Lambda environment variables
- Used for serverless Lambda function handlers

### Web Package (`packages/web`)
- **Next.js integration**: Extends Next.js ESLint rules
- **React-specific**: Includes React Hooks rules
- **Relaxed return types**: React components don't need explicit return types
- Used for the Next.js frontend application

## Running ESLint

### Lint All Packages
```bash
# From root
pnpm lint

# With auto-fix
pnpm lint:fix
```

### Lint Specific Package
```bash
# Core package
pnpm --filter @faces-of-plants/core lint

# Functions package
pnpm --filter @faces-of-plants/functions lint

# Web package
pnpm --filter web lint
```

### Auto-fix Issues
```bash
# From root
pnpm lint:fix

# Specific package
pnpm --filter @faces-of-plants/core lint:fix
```

## Common Issues and Solutions

### Import Order Violations
**Issue**: ESLint reports import order violations.

**Solution**: Run `pnpm lint:fix` to automatically reorder imports.

### Unused Variables
**Issue**: Variables are flagged as unused.

**Solution**: 
- Remove unused variables
- Prefix with `_` if intentionally unused (e.g., `_unusedParam`)

### Explicit Any
**Issue**: Using `any` type triggers warnings/errors.

**Solution**:
- Use specific types instead
- Use `unknown` for truly unknown types
- Use generics for flexible types

### Console Statements
**Issue**: `console.log` statements trigger warnings.

**Solution**:
- Remove debug console statements
- Use `console.warn` or `console.error` for intentional logging
- In Lambda functions, console is allowed

## Customization

To add package-specific rules:

1. Open the package's `eslint.config.mjs`
2. Add rules in the configuration object:

```javascript
export default [
  ...rootConfig,
  {
    files: ["**/*.ts"],
    rules: {
      // Your custom rules here
      "rule-name": "error",
    },
  },
];
```

## Ignored Patterns

The following directories are ignored by ESLint:
- `node_modules/`
- `dist/`
- `build/`
- `.next/`
- `.sst/`
- `coverage/`
- `*.config.js` files
- `*.config.mjs` files
- `*.config.ts` files
